# Compatibility and Multiplayer Contract

## Harmony surface

The mod installs two postfixes and no prefixes or transpilers:

- `CardPoolModel.GetUnlockedCards`: filters only configurable Architect Cup card IDs.
- `CardModel.ShouldGlowGold`: changes `false` to `true` only for the next valid card tracked by `VakuuTeachesUToPlayPower`.

Energy gain handling and Entomancer rewards use RitsuLib hooks. Card ownership transfer uses
`CardPileCmd.GiveToAnotherPlayer`; the mod does not patch those game methods.

## Public API

`TheArchitectCup.Api.ArchitectCupApi` is the supported integration entry point. API version 1 exposes the card
catalog, run-aware card availability, and deterministic filtering for cards generated by Burn the Mountain.
Stable string constants for every card are available from `ArchitectCupCardIds`.

```csharp
IDisposable registration = ArchitectCupApi.RegisterCrossCharacterCardFilter(
    "MyMod.exclude_special_cards",
    (context, candidate) => candidate.Rarity != CardRarity.Event);
```

Registration IDs are sorted with ordinal comparison. Filters are predicates: all registered filters must allow a
candidate. A throwing filter is ignored and logged once. Every multiplayer peer must load the same integrations and
register the same predicates; callbacks must not read local-only state or consume RNG.

The old `HandTransferPatch`, `EnergyGainPatch.Listeners`, and `CardSettingsPage.IsCardEnabled` symbols remain as
obsolete compatibility shims. New code should use the vanilla transfer command, RitsuLib player-resource hooks, and
`ArchitectCupApi` respectively.

## Determinism

- Run settings are written by the single-player process or multiplayer host and read from synchronized run data.
- Cross-character candidates and reward recipients are sorted by stable model/player IDs before RNG is consumed.
- Ping Pong does not consume gameplay RNG for visual target selection.
- Three-Legged Race energy reactions are awaited by the RitsuLib resource hook.
- Temporary card costs use the game's turn-scoped cost modifiers.

Vakuu's hand-order snapshot is current-turn transient state. Normal peers derive it from the same hand and command
history. A platform that replaces combat models in the middle of a turn without replaying that turn would need an
additional state-transfer hook; the current game API does not expose such a snapshot contract.
